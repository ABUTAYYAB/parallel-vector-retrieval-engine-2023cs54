# Benchmark Figures — Parallel Vector Retrieval Engine

Generated by `scripts/plot_results.py` from `results/benchmark_results.csv`.
Run from the project root: `src/venv/bin/python3 scripts/plot_results.py`

Hardware: Intel Core i5-1135G7 (4 cores / 8 logical threads), 8 GB DDR4-3200.
Sequential baseline: **T_seq = 0.5582 s** (N = 200,000, D = 2,048, k = 5).

---

## fig1_dimensionality_speedup.png

**What it shows:** Speedup over the sequential baseline as the embedding dimension D grows from 128 to 4,096, with N = 100,000 documents and a fixed parallelism of 4 threads or 2 ranks.

**Models plotted:** Pthreads (4t), OpenMP (4t), MPI-only (2r), Hybrid (2r × 2t), plus a dashed ideal-4× line.

**Key findings:**
- At D = 128, speedup is low (~2.5–3.3×) because the compute work per document is small and parallel overhead dominates.
- Speedup rises steeply between D = 128 and D = 1,536 as floating-point work grows.
- Above D = 1,536 the curve flattens at ~3.7–3.8× for shared-memory models (Pthreads, OpenMP, Hybrid). This plateau is the memory-bandwidth wall: multiple threads saturate the DRAM bus so additional arithmetic work does not translate into further speedup.
- MPI-only at 2 ranks consistently lags shared-memory models because intra-node MPI scatter/gather adds latency without providing an independent memory bus.

**Paper reference:** Section IV-A (Dimensionality Scaling) and Table I.

## fig2_strong_scaling_speedup.png

**What it shows:** Speedup vs. number of parallel units (threads for Pthreads/OpenMP, ranks for MPI) at fixed N = 200,000 and D = 2,048. A dashed ideal-linear line shows the theoretical maximum.

**Models plotted:** Pthreads, OpenMP, MPI-only, Ideal.

**Key findings:**
- **Pthreads** is the top performer at all thread counts. At 8 threads it reaches **5.71×** speedup — the best single-node result in the study. It also shows the most consistent scaling: near-linear up to 4 threads (≥ 95 % efficiency).
- **OpenMP** tracks Pthreads closely at 1–2 threads but diverges at 4 and 8 threads due to implicit barrier overhead and the runtime scheduler interacting with OS scheduling. At 8 threads it reaches 5.30×.
- **MPI-only** achieves near-perfect efficiency at 2 ranks (≈ 98 %) but degrades sharply at 4 and 8 ranks. On a single physical node, additional MPI ranks compete for the shared L3 cache and memory controller rather than gaining new bandwidth — the same bottleneck that limits shared-memory models, but amplified by scatter/gather overhead.
- All curves fall below the ideal line above 4 units, consistent with Amdahl's Law limiting the parallel fraction and with hardware memory-bandwidth saturation.

**Paper reference:** Section IV-B (Strong Scaling) and Table II.

## fig3_strong_scaling_efficiency.png

**What it shows:** Parallel efficiency (E = Speedup / units × 100 %) vs. parallel units for the same workload as Fig. 2. Perfect efficiency is 100 %.

**Models plotted:** Pthreads, OpenMP, MPI-only, Ideal (100 %).

**Key findings:**
- **Pthreads** maintains ≥ 95 % efficiency from 1 to 4 threads — the best sustained efficiency across all models. At 8 threads efficiency drops to ~71 % because the OS must time-multiplex 8 threads across 4 physical cores.
- **OpenMP** starts at near-100 % but falls to ~70 % at 4 threads and ~66 % at 8 threads. The implicit barrier at the end of each `parallel for` region introduces synchronisation overhead that Pthreads avoids through explicit static partitioning.
- **MPI-only** shows the steepest efficiency collapse: 98 % at 2 ranks → ~43 % at 4 ranks → ~33 % at 8 ranks. On a single node this is expected; on a true multi-node cluster with independent memory buses, MPI efficiency would remain high through additional ranks.
- This graph is the clearest evidence that **the hardware bottleneck, not the parallel model, limits scaling** above 4 units on this single-node machine.

**Paper reference:** Section IV-B and Section V (Discussion — MPI efficiency collapse).

## fig4_weak_scaling.png

**What it shows:** Wall-clock time as both the problem size (N) and number of parallel units increase together, keeping 50,000 documents per thread/rank constant. A flat line represents ideal weak scaling (constant runtime regardless of scale).

**Models plotted:** Pthreads, OpenMP, MPI-only, Ideal (constant).

**Key findings:**
- **Pthreads** is the best weak scaler: runtime is essentially flat from 1 to 4 threads (even slightly *decreasing* due to cache warming effects from larger working sets). The 38 % rise at 8 threads (4→8 logical cores on 4 physical cores) is the only significant deviation.
- **OpenMP** maintains near-constant runtime at 2 threads but rises at 4 and 8 threads. The OpenMP runtime scheduler adds barrier overhead that grows with thread count even in the weak-scaling regime.
- **MPI-only** degrades earliest and most severely: +25 % at 2 ranks, +60 % at 4 ranks, and well above ideal at 8 ranks. On a single node, MPI scatter and gather communication volume grows linearly with rank count while available bandwidth stays fixed.
- **Interpretation:** For workloads that scale with hardware (e.g., adding nodes to handle more documents in a RAG system), Pthreads and OpenMP are the right tools per node while MPI's weakness here is an artefact of single-node deployment — on a real cluster each MPI rank would own an independent memory bus.

**Paper reference:** Section IV-C (Weak Scaling) and Table III.

## fig5_execution_time_bar.png

**What it shows:** Absolute wall-clock time in seconds for every measured configuration at N = 200,000 and D = 2,048. Bars are colour-coded by model. The dashed horizontal line marks the sequential baseline.

**Configurations included:** Sequential (1×1), Pthreads (1t–8t), OpenMP (1t–8t), MPI-only (1r–8r), Hybrid (2r×1t, 2r×2t, 2r×4t, 4r×1t, 4r×2t).

**Key findings:**
- The sequential bar is the tallest at ~0.56 s. Every parallel configuration reduces runtime.
- **Pthreads 8t** produces the shortest bar (~0.10 s), confirming it as the best single-node configuration.
- **Hybrid 4r × 2t** and **Hybrid 2r × 2t** sit between MPI-only and Pthreads, demonstrating that combining both parallelism levels outperforms pure MPI on a single node.
- MPI 8r and MPI 4r bars are taller than Pthreads 4t despite the same total hardware utilisation — confirming that scatter/gather communication and memory contention across MPI processes outweigh the benefit of more ranks on a shared-memory machine.
- This figure gives the most comprehensive single-view comparison of all 17 configurations and is the primary "at a glance" summary graph.

**Paper reference:** Section IV-B (Strong Scaling), Table II, and Section V (Discussion).
